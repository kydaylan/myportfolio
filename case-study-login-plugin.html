The Brief

eventhireGroup operates 12+ bespoke B2B client portals on WordPress Multisite. Each portal requires password protection with a clean, branded login experience. The commercial plugin handling this cost £330 per site per year — £3,600+ annually across the group — and it lacked features the business actually needed.

I identified the opportunity to replace it with something better and cheaper: a custom plugin built to our exact requirements.

The Challenge

Escalating Costs

£330/year per site across 11+ sites, with costs growing every time a new client portal was added.

No Master Password

Internal staff had to know each client's individual password to access their portal. No way to use a single credential across all sites.

Broken Sessions

Sessions weren't persistent across the multisite network. Logging into one portal didn't carry over to others, disrupting internal workflow.

Generic Login UI

The customer-facing login screen was a generic form that didn't match the premium, branded portal experience clients expected.

No Customisation

No way to set per-site branding, custom logos, background colours, or login text for each client portal.

What I Built

A custom WordPress plugin called WP Site Lock that replaced the commercial solution entirely, with features it never had.

Branded Login Screens

Each client portal gets its own branded login page. The admin can upload a custom logo, set a background (solid colour, gradient, or image), and write custom login text per site. The login form uses a glassmorphism card design with backdrop blur, entrance animation, and a password visibility toggle.

Two live client portals: Tobacco Dock Food (left) and Moving Venue (right). Same plugin, different branding — each configured through the per-site admin panel.

Network-Level Master Password

A single master password managed at the network level that works across every portal. Internal staff use one credential to access any client site without needing to know individual passwords.

The network admin panel. The master password is stored at the network level and works alongside each site's own password.

Per-Site Admin Panel

Each portal has its own settings page where admins can configure the password, upload a logo via the WordPress media library, choose a background type (gradient with colour pickers and angle control, solid colour, or image upload), and set custom login page text with basic HTML support.

The per-site admin panel for Tobacco Dock, showing password management, logo upload, gradient background controls, and custom login text.

Under the Hood

The plugin follows WordPress coding standards and is structured as a proper multisite-aware plugin, not a quick hack. Key technical decisions:

  • HMAC-SHA256 cookie tokens derived from the stored password hash with separate salts for site and master passwords. Changing a password automatically invalidates all existing cookies without needing to track sessions.
  • Brute-force protection with per-IP rate limiting using WordPress transients: 10 failed attempts triggers a 15-minute lockout, scoped per site.
  • Automatic noindex/nofollow via both HTTP headers (X-Robots-Tag) and meta robots directives when a site is locked, preventing search engines from indexing protected content.
  • Nonce verification on every form submission (login and admin), capability checks on all admin actions, and full input sanitisation/output escaping throughout.
  • Admin bypass for logged-in users with manage_options capability, so WordPress admins are never locked out.
  • Clean architecture: OOP structure with separate classes for settings, network settings, protection logic, and admin UI. Frontend code only loads on the frontend; admin code only loads in the dashboard.

The plugin is network-activated across the entire multisite installation. Each sub-site gets independent settings (password, branding, enabled/disabled), while the master password is managed once at the network level. One codebase, 12+ portals, zero per-site licence fees.

How It Was Built

I wrote a detailed CLAUDE.md specification file that defined the plugin's architecture, security requirements, WordPress coding standards, and the expected file structure. This acted as the brief for Claude Code to build from.

The spec covered everything: the cookie authentication approach, the multisite settings separation, the admin UI requirements, security rules (nonces, capability checks, input sanitisation, output escaping), performance constraints (don't load admin code on the frontend), and the activation/deactivation behaviour.

From there, I used Claude Code to develop the plugin to spec, then tested and iterated across all 12+ portals until it worked correctly in every scenario: site password, master password, admin bypass, rate limiting, and branded login pages with different configurations.

This is how AI-assisted development works in practice. I identified the problem, designed the solution, wrote the technical specification, and directed the build. The AI wrote the code to my architecture. I tested, iterated, and deployed. The strategic thinking and the specification writing are the real skills — the code generation is the execution method.

What I Learned

The cost of commercial plugins doesn't always reflect their value. A bespoke solution built in days saved thousands annually while adding features the commercial option never offered: per-site branding, a network master password, proper rate limiting, and automatic search engine blocking.

The real skill was identifying the opportunity. Recognising that the business was paying £3,600/year for something that could be replaced with a better, custom solution required understanding both the technical landscape and the business cost. Writing a clear, detailed specification was what made the AI-assisted build possible — without that spec, the output wouldn't have met WordPress coding standards or handled the multisite edge cases correctly.
