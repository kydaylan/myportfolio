The Brief

eventhireGroup needed bespoke online portals for its highest-value B2B account customers. Each portal needed to look and feel unique to the client, display individually negotiated pricing, and be maintainable without duplicating effort across dozens of separate WordPress installations.

The Challenge

Delivering a scalable portal system for multiple B2B clients, each with unique requirements, without creating a maintenance nightmare.

Unique Branding

Each B2B client needs their own branded portal with unique appearance, colours, and layout.

Individual Pricing

Pricing is individually negotiated per client — every portal shows different prices for the same products.

Maintenance at Scale

Maintaining 12+ separate WordPress installations would be unmanageable for a one-person digital team.

Rapid Onboarding

New portals need to be spun up quickly when new account clients are onboarded.

Bespoke Client Requirements

Every client has different needs: some require two-step login flows, others want prices hidden, some need venue-specific delivery locations. Each requires custom implementations while staying within the shared theme.

A WordPress Multisite architecture that powers 12+ bespoke client portals from a single installation:

  • Each portal has individually configurable appearance (branding, colours, layout) while running on a shared theme
  • Custom pricing per portal — the same product can show different prices to different clients, with the option to hide prices entirely
  • Per-client custom functionality: two-step login flows, venue-specific delivery locations, bespoke checkout requirements — implemented through ACF fields, custom PHP, or theme-level changes depending on the requirement
  • Shared product catalogue maintained once, displayed everywhere
  • New portals can be created and configured rapidly, with each iteration building on the shared system without breaking existing portals
  • Integrated with the custom login plugin for secure, branded access
  • Part of the broader "Premier Partners" service alongside the AI product visualiser

Architecture

WordPress Multisite was the right choice because it allows one core installation with shared plugins, themes, and a product database, while each sub-site (portal) can have its own configuration, appearance settings, and pricing rules.

Product updates happen once and propagate everywhere, while each client sees their negotiated prices and branded experience. The alternative — maintaining 12+ separate sites, updating products in each one individually, and managing separate hosting for all of them — would have been unsustainable for a one-person team.

The real complexity is in the per-client customisations. Each portal can require different business logic, and every change has to work for that specific client without breaking the other 11+ portals sharing the same system.

Shared Customisations (All Sites)

All portals share a base layer of WooCommerce modifications that reflect how the hire business operates:

  • "Shipping" renamed to "Transport" across checkout, cart, emails, and order totals — because hire companies deliver, they don't ship
  • "Billing details" renamed to "Order Request Details" and order confirmation language changed throughout ("order request" not "order") to reflect the enquiry-based workflow
  • Custom quantity controls with plus/minus buttons replacing the default WooCommerce number input
  • UK date formatting via a custom shortcode for hire start/end dates in emails
  • Dynamic product category grid built as a custom shortcode that pulls WooCommerce categories with thumbnails into a responsive bento layout

Per-Site Customisations

Individual portals then get site-specific logic layered on top, targeted by blog ID:

  • Hidden pricing for guests — some portals hide all prices until logged in, showing "Login to view price" on product pages and nothing on category pages. Cart and checkout are also blocked for guests.
  • Site-specific order messaging — e.g. Tobacco Dock's on-hold message references the client's account manager rather than using the generic WooCommerce text
  • Custom checkout redirects — e.g. Goodwood's portal redirects to a custom "order-request" URL slug instead of the default WooCommerce checkout path
  • Per-client hero imagery, colour schemes, and navigation configured through the theme settings per sub-site

What I Learned

The key decision was choosing WordPress Multisite over separate installations. It's a trade-off: Multisite adds some complexity in how plugins and themes interact across the network, but for this use case — dozens of portals that share a product catalogue but need individual pricing and branding — it was clearly the right architecture.

The alternative would have been maintaining 12+ separate sites, updating products in each one individually, and managing separate hosting for all of them. That would have been unsustainable for a one-person team.

The most technically challenging part is the ongoing per-client customisation. Every client has different requirements — some need two-step logins, others want prices hidden, some need specific venue delivery locations with custom checkout flows. Each of these needs implementing through a mix of ACF fields, custom PHP, or theme-level changes, and every change has to work for that client without breaking the other portals sharing the same system. It's a constant balancing act between bespoke client needs and system stability.
